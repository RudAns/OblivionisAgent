# 🧩 循环 · Perforce(P4) 写改技能模板

给「会真的动代码、但**只改不提交**」的循环用。把下面「---」之间的内容**贴进一个「🧩 技能」节点**，
连到被「🔁 循环」节点驱动、且工作目录是一个 **P4 工作区(workspace)** 的会话上。

核心约定一句话：**自动 `p4 edit`/`p4 add` → 改 → 留在 pending changelist 里 → 永不 `p4 submit`，提交永远由人来。**

> 为什么不用 worktree 隔离？那是 git 的玩法；P4 用的是 workspace + pending changelist，
> **pending CL 本身就是天然的审查边界**——循环把活干到 CL 里，人审完再点提交，安全且贴合 P4 工作流。
> （游戏资产动辄上 GB，也不适合每轮另开一个 workspace。）

---

## 你正处在一个「循环」里（会反复触发，多轮推进）

- 每轮**只推进一小步**，做完即停，等下一轮；稳比快重要。
- **全部完成时，单独回复一行**：`[[DONE]]`（以循环节点设的「完成标记」为准）。没全做完别回它。
- 跨轮记忆靠工作目录的 `STATE.md`：每轮开头先读它（上下文可能被重置过），结束把进度写回。

## 改文件前：先 `p4 edit` / `p4 add`（这是 P4 的硬规矩）

- 改**已在版本库**的文件前，先 `p4 edit <文件>` 把它 checkout 出来（否则文件只读、改不动）。
- **新增**文件，先建好再 `p4 add <文件>`。
- 删除文件用 `p4 delete <文件>`，不要直接 rm。
- 改完用 `p4 opened` 看一眼这次都动了哪些文件，确认范围对。

## 把改动归到一个**专用 pending changelist**（别丢进 default）

- 第一轮先建一个带说明的 changelist，例如：
  - `p4 change -o` 拿模板 → 填好 `Description:`（写清这次循环要干的事）→ `p4 change -i` 提交模板得到一个 **CL 号**。
  - 之后所有 `p4 edit/add/delete` 都带上 `-c <CL号>`，让改动都归到这个 CL。
- 把这个 **CL 号写进 `STATE.md`**，后续每轮都复用它，别每轮新建。
- 收尾时跑 `p4 opened -c <CL号>` 把这次 CL 里的文件清单列进你的总结。

## ⛔ 绝对不要做的事

- **绝不 `p4 submit`**。提交是人的决定——人会 review 这个 pending CL 再自己提交。
- 不 `p4 revert`（除非主人明确要求回退某个改动）——别擅自丢掉已做的工作。
- 不碰别的 changelist、不动不属于本任务的文件。
- 不改 P4 配置（`p4 set` / client spec / typemap 等）。

## 造 → 检（maker-checker）

- **造**：做当前这一小步，按上面规矩 `p4 edit` 后再改。
- **检**：做完立刻自查（编译/类型/测试/`p4 diff` 看改动是否如预期）；复杂时用 `Task` 委派给「检查者」子代理独立挑错。
  - 没过 → 在 `STATE.md` 记下问题，下一轮先修。
  - 过了 → 标记完成，进入下一步。

## 安全（L1 / 审批）

- 默认偏**只读 + 报告**；`p4 edit`、改文件、跑命令这类需授权的操作会自动弹**审批卡**给主人——正常发起、让主人把关即可。
- **绝不**为了凑 `[[DONE]]` 假装完成、跳过验证、或偷偷 submit。

---

## 配套：循环节点怎么配

- **任务 prompt**：说清「在这个 P4 工作区里要改什么、怎么算完、要不要顺手写测试」。
- **停止方式**：选「出现完成标记即停」，完成标记 `[[DONE]]`。
- **最多轮数**：兜底刹车，按任务大小给（如 5～15）。
- **每 N 轮重置上下文**：长任务建议设（如 3～5），配合 `STATE.md` 与上面记下的 CL 号续接。
- **详细报告**：想让它在审核群简报之外再产出一份「这次 CL 改了什么、为什么」的存档，把循环节点的「详细报告」设成 `md` 或 `html`。
- 收尾后由**主人**在 P4 客户端里 review 这个 pending CL，确认无误再 **submit**。
