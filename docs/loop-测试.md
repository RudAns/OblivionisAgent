```

```
然后双击根目录 **`rebuild-deploy.bat`**（重打 bridge + desktop → 关掉运行中的 app → 覆盖便携版 → 重启）。

## 三、测试用例（任务 prompt 直接复制）

### 例 1 · 纯只读机制测（推荐先跑这个）

**任务 prompt：**
```
我们分多轮完成一件事：把当前工作目录下的子文件夹逐个列出来，每个配一句用途推测。
规则：每一轮只处理「下一个」还没处理过的文件夹，报它的名字 + 一句它大概是干嘛的。
当所有子文件夹都处理完了，才单独回复一行：[[DONE]]（在那之前绝不要回这一行）。
现在开始处理第一个。
```
**设置**：停止方式 = `出现完成标记即停`，完成标记 = `[[DONE]]`，最多轮数 = `5`。
**预期**：一轮报一个文件夹 → 处理完所有的回 `[[DONE]]` 停；不够 5 个提前停，超过 5 个到 5 轮被刹车停。

---

### 例 2 · 最稳看到「多轮 + 刹车」

有些模型会一轮干完就直接 `[[DONE]]`（只跑 1 轮，也对）。想稳稳看到循环转起来：

**任务 prompt：**
```
每一轮用一句话，说一条关于当前这个项目的观察（代码结构 / 技术栈 / 可改进点都行），每轮换一条、别重复。
```
**设置**：停止方式 = `固定跑满轮数`，最多轮数 = `3`。
**预期**：老老实实跑满 3 轮后停——最直观地验证「循环 + 刹车」。

---

### 例 3 · 更像真实活（只读审阅）

**任务 prompt：**
```
分多轮把 src/ 目录下的源码文件逐个快速过一遍，找明显问题（空的 catch、可疑 TODO、硬编码的密钥/地址等）。
每一轮只看「下一个」还没看过的文件，列出该文件发现的问题（没问题就写"无"）。
所有文件都过完后，单独回复一行：[[DONE]]。
```
**设置**：停止方式 = `出现完成标记即停`，最多轮数 = `10`。

---

### 例 4 ·（可选）每 N 轮新鲜上下文

> 这条需要该会话**设了 baseSessionId**（它靠从 base 重新 fork 来「刷新」——没 base 就没东西可 fork）。

**任务 prompt：**
```
分多轮整理这个项目的「模块清单」。每轮处理一个模块：在工作目录的 STATE.md 里追加「模块名 + 一句职责」，并在 STATE.md 顶部维护「已完成 / 待办」。
每轮开头先读 STATE.md 看进度再继续。全部模块整理完后，单独回复一行：[[DONE]]。
```
**设置**：停止方式 = `出现完成标记即停`，最多轮数 = `8`，**每 N 轮重置上下文 = `2`**。
**预期**：每 2 轮后重新 fork（服务日志里有「重置上下文」），靠 STATE.md 续接、不丢进度。

## 四、看这几样对不对

- 打开那个会话的**终端 / 转录**：看它被**连发多轮**；
- 命中 `[[DONE]]` 或满轮数就**停**（不空转）——检视里有实时轮次显示；
- `C:\Users\chenjingwen\.oblivionis\loop-logs\` 下生成了一份 run-log（`<节点名>-<时间>.md`）；
- 要看汇总**发到飞书群**：需连着飞书 + 配了 Home Chat 或在节点填「投递群 chatId」；**纯测机制不需要飞书**。

要点：任务里的「每轮只做一个/一步」是关键，否则它一口气做完就只有 1 轮；`[[DONE]]` 必须**单独成行**且和节点「完成标记」字段完全一致。
